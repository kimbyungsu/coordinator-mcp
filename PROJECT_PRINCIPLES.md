# Project Principles — Coordinator MCP 개발 원칙

> 이 문서는 이 프로젝트를 설계하고 구현하는 **AI 모델(구현 모델 / 검증 모델)**이 지켜야 할 원칙이다.
> 제품이 최종 사용자에게 제공하는 런타임 규칙(RUNTIME.md)이나 에이전트 주입 문장(PROMPT_CONTRACT.md)과는 분리된다.
>
> 한 줄 요약: 이 문서의 독자는 이 MCP를 **만드는** 모델이지, 이 MCP가 **구동하는** 에이전트가 아니다.

---

## 1. 설계 후퇴 감지 규칙

다음 제안은 즉시 후퇴안으로 분류하고 채택하지 않는다.

```
[후퇴 패턴 목록]
- "우선 수동으로 시작하고 나중에 자동화"
- "사용자가 다음 에이전트에게 알려주면 됨"
- "초기엔 터미널 두 개 켜두고 교대로 실행"
- "일단 이렇게 해두고 나중에 개선"
- "초기 버전이니까 반자동이 현실적"
- "MVP니까 수동 전환도 허용"
```

이 패턴이 등장하면 **구현 편의에 의한 제품 정체성 훼손**으로 간주한다.  
이 프로젝트의 핵심 명제("사람이 턴을 중계하지 않는다")를 협상하는 순간 이 MCP의 존재 이유가 사라진다.

---

## 2. 구현 관성 방지

AI 모델이 설계 작업 중 빠지기 쉬운 관성 패턴:

| 관성 패턴 | 실제 발생 형태 | 올바른 대응 |
|-----------|--------------|------------|
| 편의 우선 | 어려운 자동화 대신 "반자동도 괜찮지 않을까" 제안 | 자동화 가능성을 먼저 탐색하고, 불가 시 사유 명시 |
| 계층 혼동 | 제품 런타임 문서에 개발 가이드라인 기재 | 문서 독자를 항상 먼저 확인: 이 문서는 누가 읽는가? |
| 후진 정당화 | "초기 버전이니까" 논리로 금지선 우회 | RUNTIME.md Section 1 "자동화 최소 요건" 재확인 |
| 조용한 범위 확장 | 요청 없는 기능 추가나 "개선" 제안 | 요청된 작업만 수행, 추가 제안은 별도 명시 |

---

## 3. 문서 계층 원칙

이 프로젝트의 문서는 세 계층으로 분리된다.

```
[계층 1: 개발 원칙] — 이 문서 (PROJECT_PRINCIPLES.md)
  독자: 이 MCP를 만드는 AI 모델 / 설계자
  내용: 설계 후퇴 방지 규칙, 구현 관성 방지, 문서 계층 원칙

[계층 2: 런타임 설계] — RUNTIME.md
  독자: 이 MCP를 구현하는 개발자
  내용: Turn Dispatcher 구조, Runtime A/B 선택 기준, Dispatcher 루프 설계, 구현 로드맵

[계층 3: 에이전트 주입 문장] — PROMPT_CONTRACT.md
  독자: Coordinator가 각 에이전트에게 주입할 실제 문장을 작성하는 개발자
  내용: Implementer / Verifier / Discussion 역할별 주입 원문, 금지 표현, 권장 표현

[제품 설계 전반] — DESIGN.md
  독자: 제품 전체 구조를 이해하려는 모든 참여자
  내용: 상태 기계, 모드, Issue Ledger, Obligation 생애주기, 체크포인트 복원
```

계층 간 내용이 섞이면 즉시 분리한다.  
"이 내용은 어느 계층 독자를 위한 것인가?"를 기준으로 판단한다.

---

## 4. 검증 원칙 (이 프로젝트 내 검증 모델용)

이 문서의 규칙은 DESIGN.md / PROMPT_CONTRACT.md의 Verifier 규칙과 별개다.  
이 섹션은 이 프로젝트 설계 작업 자체를 검증하는 모델이 지켜야 할 원칙이다.

```
[검증 시 확인 항목]
1. 제안된 설계가 RUNTIME.md Section 1 금지선을 위반하는가?
2. 문서 내용이 올바른 계층에 기재되어 있는가?
3. "편의상 수동" 논리가 합리화되어 있지 않은가?
4. 자동화 최소 요건(MVP 기준)을 실제로 충족하는 설계인가?
```

```
  [ ] git 커밋 여부를 확인하거나 직접 커밋을 수행한다
```
